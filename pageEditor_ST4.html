Sitemap Page

Technical Description

The Sitemap page provides a visual overview of the entire Blue Horizon website structure using SVG (Scalable Vector Graphics). The page is built using the shared HTML template, linking to the team's Style.css for consistent styling across the site.

The SVG element uses a viewBox="0 0 1000 480" attribute with width="100%" and height="auto" set via CSS, which makes the diagram fully responsive and scalable across different screen sizes without losing clarity. All nodes are drawn using SVG <rect> elements with rounded corners using the rx attribute, and connected using <line> elements styled with custom stroke colours and widths. A dashed line using stroke-dasharray visually separates the navigation page tier from the content page tier, making the hierarchy clearer to the user.

Every node is wrapped in an SVG <a> anchor element with tabindex="0" for keyboard accessibility. Each anchor contains a <title> element for screen readers. A role="img" and aria-label are applied to the entire SVG element. Hover and focus effects are handled via CSS targeting a.smap-node:hover rect and a.smap-node:focus-visible rect, which change the fill to lightblue. The current page (Sitemap) is styled with a lightblue fill and a "YOU ARE HERE" label. A colour-coded legend below explains each node colour.

HTML Structure

  • SVG element with viewBox: Creates a scalable coordinate system for the diagram that responds to any screen size without losing proportions.
  • <rect> elements: Used to draw each page node as a rounded rectangle. The rx attribute rounds the corners for a modern appearance.
  • <line> elements: Connect the nodes to show relationships. Dashed lines use stroke-dasharray to visually separate the two tiers.
  • <a> anchor elements with tabindex: Wrap each node to make it clickable and keyboard accessible.
  • <title> elements: Provide accessible descriptions for each node for screen reader users.

CSS Techniques

  • width: 100% and height: auto: Makes the SVG fully responsive — it scales proportionally to fit any container width.
  • a.smap-node:hover rect and :focus-visible: Changes the node fill colour on hover and focus, providing clear visual interaction feedback.
  • stroke-dasharray: Creates dashed connector lines to visually separate the navigation tier from the content page tier.
  • CSS animation @keyframes fadeIn: Not used on sitemap — used on profile page instead.

Accessibility

The SVG includes role="img" and a detailed aria-label describing the full diagram to screen reader users. Each clickable node has a <title> element inside its anchor group providing a meaningful description of where that link leads. All nodes have tabindex="0" so keyboard users can navigate between them using the Tab key. Focus states are visible — when a node is focused the rectangle changes colour in the same way as the hover state, implemented using the CSS :focus-visible pseudo-class. Text labels are visible directly in the diagram so no information is conveyed by colour alone.

User Profile Page

Technical Description

The User Profile page collects information from the user through JavaScript prompt() dialogs as required by the coursework specification — not HTML forms. The page uses a two-column Flexbox layout: a control panel on the left containing buttons and a progress bar, and a profile display on the right that builds progressively as the user completes each step.

All user data is stored in a single JavaScript object called profileData, with each property initialised to null. A separate object stepDone tracks which of the three steps are complete. The profile-building process has three steps handled by the runStep(stepNumber) function. Step 1 collects basic details (name, username, location), Step 2 collects ocean interests (favourite marine topic, visit frequency, biggest concern), and Step 3 collects actions and pledges (pledge action, people to tell, commitment level). Pressing Cancel on any prompt skips it — the value stays null and that row is hidden using the toggleRow() function which sets display:none on the row.

The progress bar is updated by updateProgress(), which counts how many of the 9 individual prompts have been answered and calculates a percentage out of 9. This means skipping prompts genuinely reduces the completion percentage. The progress bar uses a CSS gradient fill with a smooth transition: width 0.5s ease animation. Profile sections are hidden initially and revealed by adding the visible CSS class using classList.add(), with a @keyframes fadeIn animation on reveal. Revisit buttons allow users to re-run any step at any time, updating the profile and progress bar with the new data.

HTML Structure

  • Two-column Flexbox layout: Separates the control panel from the profile display, keeping the interface clean and organised.
  • progress-track and progress-fill divs: Create the custom progress bar — the fill width is controlled dynamically by JavaScript.
  • profile-section divs: One per step, hidden initially using CSS and revealed progressively as steps complete.
  • profile-row divs with id attributes: Each row represents one prompt answer. The toggleRow() function shows or hides each row based on whether the user answered that prompt.
  • ARIA attributes on progress bar: role="progressbar", aria-valuemin, aria-valuemax, and aria-valuenow updated dynamically by JavaScript.

CSS Techniques

  • display: flex with flex-wrap: wrap: Creates the two-column layout that collapses to a single column on smaller screens.
  • @keyframes fadeIn: Fades in each profile section as it becomes visible, making the progressive reveal feel smooth rather than abrupt.
  • transition: width 0.5s ease: Animates the progress bar fill as it grows with each completed prompt.
  • display: none / display: flex: Controls visibility of profile rows — skipped prompts are hidden completely rather than showing a placeholder.

Accessibility

The progress bar element uses ARIA attributes — role="progressbar", aria-valuemin, aria-valuemax, and aria-valuenow — which are updated dynamically by JavaScript as the user completes prompts, ensuring screen readers can announce the current completion percentage. All buttons have meaningful visible labels. Instructions within each prompt are written clearly with examples so users understand what input is expected. The active nav link has aria-current="page".

Content Page — Marine Technology

Technical Description

The Content page covers Marine Technology and its role in SDG 14: Life Below Water. It is structured as a long-form article using the HTML <article> element, divided into clearly labelled sections using <h2> headings. Topics covered include Autonomous Underwater Vehicles, ROVs, satellite technology, AI and machine learning, environmental DNA, ocean sensor networks, 3D mapping, and innovative conservation techniques.

The page uses a glassmorphism-style container with background: rgba(0,0,0,0.7) and backdrop-filter: blur(10px), giving it a frosted glass appearance consistent with the team's ocean aesthetic. A full-width hero image sits at the top followed by an <hr> element styled as a gradient line using background: linear-gradient(to right, transparent, skyblue, transparent). All images include descriptive alt text with source references.

HTML Structure

  • <article> element: Used as the semantic container for the content as it is a self-contained independent piece of writing, which helps search engines and screen readers identify the primary focus of the page.
  • Heading hierarchy: One <h1> for the page title followed by <h2> headings for each section, providing a logical structure for both readability and screen reader navigation.
  • <ul> lists: Used to break up dense text into scannable bullet points, particularly for the AI capabilities section.
  • <strong> elements: Highlight critical statistics and key terms throughout the article.

CSS Techniques

  • rgba(0,0,0,0.7) and backdrop-filter: blur(10px): Creates the glassmorphism container effect — semi-transparent with a blurred background that keeps the ocean theme visible while keeping text readable.
  • linear-gradient HR: Styles the section divider as a soft gradient line from transparent to skyblue, acting as a visual break between sections without a harsh border.
  • object-fit: cover: Applied to the hero image to ensure it fills its container without distortion on any screen size.
  • transition and box-shadow on .back-btn: Provides smooth hover feedback on the Back to Home button.

Accessibility

All images include meaningful alt text that describes the image content and its source URL. The article uses a logical heading hierarchy — one <h1> for the title followed by <h2> for each section — which supports screen reader navigation. Text uses line-height 1.8 and font-size 18px for comfortable reading. The "Back to Home" button has a clear descriptive label and visible hover feedback.

Challenges and Lessons Learned

Challenges Faced

1. Learning SVG from scratch

I had never used SVG before this project. Creating the sitemap required learning how to draw shapes using <rect> and <line> elements, position them using coordinate attributes, and make them interactive with anchor elements and CSS hover states. The hardest part was calculating x and y positions so the connector lines joined nodes correctly. I solved this by sketching the layout on paper first and working out the coordinates manually before writing the code.

2. GitHub conflicts when pushing files

Several times when I tried to push my files to the team repository I received merge conflict errors because my teammates had pushed changes at the same time. I did not know how to resolve these at first. I learned that running git pull --rebase before pushing merges the remote changes into my local version cleanly, avoiding conflicts. This taught me the importance of always pulling before starting a push session.

3. JavaScript prompt() showing the IP address

When testing the profile page locally using Live Server, the browser's prompt dialog showed 127.0.0.1:5500 at the top. I initially thought this was a bug in my code. After researching I found this is a built-in browser security feature that cannot be removed — it tells the user which website is requesting their input. I learned this is expected behaviour and that the specification specifically requires prompt(), so this is the correct approach.

4. Understanding DOM manipulation

The User Profile page required making elements appear and disappear dynamically without reloading the page. I found it confusing at first to understand how JavaScript could change what was visible in the HTML. Through building the profile page I learned how document.getElementById() selects elements, how classList.add() and classList.remove() apply and remove CSS classes, and how style.display shows or hides elements. By the end of the project I felt much more confident using the DOM to build interactive pages.

Lessons Learned

Plan before you code. For the SVG sitemap, sketching the layout on paper before writing any code saved a lot of time. Having the coordinates planned out meant I could write the SVG elements confidently without trial and error.

Version control is essential. The GitHub conflicts taught me that git pull should always come before git push. Working in a team means the repository is always changing and keeping your local copy up to date prevents most conflicts from happening in the first place.

Read the specification carefully. The prompt() IP address issue showed me the importance of understanding why a tool behaves in a certain way before assuming it is broken. The specification required prompt() specifically, and understanding the browser's security model helped me explain the behaviour rather than trying to fix something that was not broken.
